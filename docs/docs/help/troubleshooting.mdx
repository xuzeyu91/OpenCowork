# Troubleshooting

先按症状定位，再回到具体页面看配置细节。OpenCowork 是 local-first 桌面应用，大多数问题都能从依赖、模型配置、workspace、channel 凭据和本地状态这几条线排查。

## App does not start

<Runbook>
  <RunbookStep title="Check Node.js">
    确认 Node.js 版本满足项目要求，当前项目要求 Node.js 18 或更高。
  </RunbookStep>
  <RunbookStep title="Reinstall dependencies">
    重新安装依赖并执行 native module rebuild。
  </RunbookStep>
  <RunbookStep title="Run type check">
    如果应用无法编译，先用类型检查确认不是 TypeScript 或路径别名问题。
  </RunbookStep>
</Runbook>

<CommandGroup title="App checks">
  <CommandItem label="Install" command="npm install" />
  <CommandItem label="Rebuild" command="npm run postinstall" />
  <CommandItem label="Types" command="npm run typecheck" />
</CommandGroup>

## Docs build fails

文档站是独立 Next.js 应用，先进入 `docs/` 再检查。

<CommandGroup title="Docs checks">
  <CommandItem label="Enter" command="cd docs" />
  <CommandItem label="Types" command="npm run types:check" />
  <CommandItem label="Build" command="npm run build" />
</CommandGroup>

常见原因：

- MDX frontmatter 缺少 `title` 或语法错误。
- `meta.json` 引用了不存在的页面。
- 自定义 MDX 组件没有在 `src/mdx-components.tsx` 注册。

## Model calls fail

<Runbook>
  <RunbookStep title="Check provider">
    确认 provider preset 已启用，并且当前 session 使用的是预期模型。
  </RunbookStep>
  <RunbookStep title="Check credentials">
    检查 API key、OAuth 或 compatible endpoint 是否可用。
  </RunbookStep>
  <RunbookStep title="Switch provider">
    临时切换到另一个 provider，判断问题来自单个服务还是 OpenCowork 配置。
  </RunbookStep>
</Runbook>

## Channel integration fails

<Runbook>
  <RunbookStep title="Credentials">
    先确认 channel credentials、bot token、webhook 或登录态。
  </RunbookStep>
  <RunbookStep title="Platform permissions">
    再检查平台侧 webhook、socket、机器人权限、群权限和回调地址。
  </RunbookStep>
  <RunbookStep title="OpenCowork config">
    最后回到对应 channel 页面核对 OpenCowork 内部配置和运行状态。
  </RunbookStep>
</Runbook>

## Tool result looks stale

- 确认当前 session 绑定的 workspace。
- 检查文件是否由外部进程改动。
- 重新打开 preview 或刷新对应面板。
